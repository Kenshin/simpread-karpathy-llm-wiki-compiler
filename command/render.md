# render.md: 排版模板系统协议

### 1. Role: 排版引擎

你是 `--lite` / `--kami` 排版模板系统的唯一入口。任何命令产出内容后附加 `--lite` 或 `--kami` 时，最终的 HTML 排版都由本协议负责。

支持两种排版模式：
- **`--lite`（默认）**：轻量排版方案，零依赖，基于 `tools/template/lite.html`
- **`--kami`**：通过 kami 设计语言排版，需要安装 kami skill，与 `--lite` 互斥

### 2. 入口路由：谁触发排版？

`--lite` / `--kami` 是跨环境通用修饰符（见 `AGENT.md` 8.3），可被以下任意命令触发：

| 入口命令 | 环境 | 内容来源 | 输出文件名 |
|----------|------|----------|-----------|
| `/render {主题} --lite` | Wiki | `wiki/{主题}.md` | `{主题}.html` |
| `/report {主题} --lite` | Wiki | AI 基于 wiki/ 生成的简报 | `{主题}-简报.html` |
| `/ask {问题}` | Wiki | AI 基于 wiki/ 生成的回答 | `{主题}.html`（AI 提取主题名） |
| `MCP 搜索 --lite` | MCP | AI 搜索/生成的动态内容 | `{主题}-{关键词}.html` |

**路由规则**：
- `/render` 是**直接排版**：读取已有 `wiki/{主题}.md`，执行内容编译 + 排版
- `/report`、`/ask` 是**生成后排版**：先按各自协议（`report.md`、`ask.md`/`qa.md`）生成结构化内容，再交给本协议排版
- MCP 环境是**搜索后排版**：先按 `simpread-mcp-helper` 执行搜索，再交给本协议排版
- 所有入口最终都走本协议的**排版模板规范**（`--lite` 走 §5.1，`--kami` 走 §3.2 + §6）

### 3. 排版模板规范

#### 3.1 `--lite` 模板规范

**模板加载**：
1. 读取 `tools/template/lite.html` 作为排版骨架
2. 读取 `tools/template/lite-diagrams.md` 作为图表组件规范
3. 若模板文件不存在，**终止并提示**

**内容编译规则**（适用于所有入口）：

1. **Header**：
   - eyebrow：从主题/文件夹名推导分类前缀（`标签@观察_新消费_黄金` → "新消费观察 · 黄金专题"），无分类前缀则跳过
   - **标题**：清洗 `标签@` 前缀，提取有意义部分
   - 副标题/Lead：从内容摘要提取
   - meta：作者从当前项目顶层文件夹名称获取（如 `simpread-kb` → "SimpRead KB"），日期自动生成

2. **Metric Cards**：从正文"核心数据"表格提取 4 个关键指标，无数据则跳过

3. **TOC**：提取所有 h1 章节标题，生成锚点链接

4. **知识图谱**：
   - 默认：将 Mermaid 语法转为内联 SVG
   - `--mermaid` 模式：用 `<pre class="mermaid">` + Mermaid CDN 加载
   - 复杂度降级：超过 20 个节点时自动降级为 Mermaid CDN

5. **SVG 图表**：从 `tools/template/lite-diagrams.md` 的 8 种组件中选取适合的图表类型（无条件自动选用）

6. **快照链接**：Wiki 中的 `[本地快照](http://localhost:7026/reading/4873)` 排版为 `<a class="snapshot" href="http://localhost:7026/reading/4873" target="_blank" rel="noopener">4873</a>`——链接文本仅保留快照 ID 数字。所有非锚点链接（快照、原文引用、Sources 表中的链接等）都必须在 HTML 中直接写入 `target="_blank" rel="noopener"`，TOC 的 `#` 锚点链接除外

7. **Sources 映射表**：完整保留

8. **暗色模式**：toggle 按钮必须存在

#### 3.2 `--kami` 模板规范

kami 作为已安装 skill，其模板资产（`long-doc.html`、`one-pager.html` 等）始终可用。

1. **模板来源唯一**：排版 HTML 必须基于 kami skill 目录内的模板文件生成
2. **严禁借用外部文件**：不得使用 `outputs/` 或其他目录下的已有 HTML 文件作为模板
3. **不得从零编写**：不得在未加载 kami 模板的情况下自行编写 HTML
4. 读取 kami `SKILL.md`，遵循其排版规范
5. 文档类型：默认 `long-doc`，除非用户通过 `--type` 指定

### 4. 修饰符参数

`--lite` 和 `--kami` 可附加在任何产出内容的命令之后：

```
/render [主题] [--lite|--kami] [--mermaid] [--pdf] [--svg] [--type <类型>]
/report [主题] [--lite|--kami] [-m]
/ask [问题] [--lite|--kami]
<MCP 搜索命令> [--lite|--kami]
```

| 参数 | 必填 | 说明 |
|------|------|------|
| `--lite` | 否 | 使用轻量排版方案（默认，与 `--kami` 互斥） |
| `--kami` | 否 | 使用 kami 排版方案（需安装 kami skill） |
| `--mermaid` | 否 | 仅 `--lite` 模式：知识图谱使用 Mermaid CDN 渲染 |
| `--pdf` | 否 | 同时生成 PDF（需 WeasyPrint，仅 `--kami`） |
| `--svg` | 否 | Mermaid 转手绘 SVG（仅 `--kami`） |
| `--type <类型>` | 否 | 覆盖文档类型（仅 `--kami`）。可选：`long-doc` / `one-pager` / `equity-report` / `portfolio` / `slides` / `letter` |

**互斥规则**：`--lite` 与 `--kami` 不能同时使用。不指定时默认走 `--lite`。
**`--kami` 默认行为**：无 `--type` 时强制使用 **long-doc** 模板。

### 5. 工作流（所有入口共用）

以下是排版的通用流水线。不同入口的差异仅在 **Step 1（内容获取）**，后续步骤完全共用。

#### Step 1: 内容获取（按入口分发）

**入口 A：`/render {主题}`（直接排版）**

1. 拼接路径 `wiki/{主题}.md`
2. **检查文件是否存在**：若文件不存在，**终止并提示**：
   > 未找到 `wiki/{主题}.md`。请确认主题名称或先生成该 Wiki 条目。
3. **检查同名文件（提前）**：在读取文件内容前，先检查 `outputs/` 下是否已存在以下文件：
   - `outputs/{主题}.html`
   - `outputs/{主题}.kami.html`
   - `outputs/{主题}.lite.html`

   **若存在同名文件**：**暂停并输出状态信息，使用交互式选项提示用户**：

   ```
   outputs/{主题}.html 已存在。请选择输出方式：
   1. 覆盖 咖啡.html
   2. 覆盖 咖啡.lite.html
   3. 覆盖 咖啡.kami.html
   4. 按模式命名
   5. 自定义文件名（由用户指定输出路径）
   6. 取消（保留原文件，不生成新文件）
   ```

   **选项说明**：
   - **覆盖 {文件名}**：替换指定的已有文件（显示文件大小）
   - **按模式命名**：使用 `{主题}.kami.html` 或 `{主题}.lite.html`
   - **自定义文件名**：由用户指定输出路径
   - **取消**：不生成，保留原文件

   **若用户选择"取消"**：**立即终止**，不读取文件内容，避免无效的 token 消耗。

   **若用户选择其他选项**：继续执行后续步骤。

   **若不存在同名文件**：文件名命名为 `{主题}.html`

4. 读取文件内容
   - **大文件完整读取协议**：
     - 先获取文件总行数：`wc -l wiki/{主题}.md`
     - **总是分块读取**：无论文件大小，都采用分块读取策略，确保 100% 完整性
     - 分块策略：每次读取 500 行，使用 `offset` 和 `limit` 参数
     - **禁止跳读**：不得因"内容相似"或"不重要"而跳过任何块
     - **完整性验证**：读取完成后，核对首尾行号和章节总数
   - **示例流程**（大文件）：
     ```
     # 1. 获取总行数
     wc -l wiki/标签@观察_商业历史_外卖大战.md  # 输出：781

     # 2. 分块读取（总是分块）
     Read wiki/...md (offset=0, limit=500)     # 块 1：行 1-500
     Read wiki/...md (offset=500, limit=500)    # 块 2：行 501-781

     # 3. 完整性验证
     - 首行：# 标签@观察_商业历史_外卖大战
     - 末行：| 编号 | 快照数量 | ...
     - 章节总数：22 个 ## 标题
     ```

#### Step 2: 加载 Kami 规范

1. 读取 kami `SKILL.md`，遵循其排版规范
2. 文档类型：默认 `long-doc`，除非用户通过 `--type` 指定
3. 语言：中文 → 使用中文模板路径

#### Step 3: 内容编译

**3.0 内容模式判断（Content Mode Decision）**

在开始编译前，必须先判断内容模式。判断依据：

| 指标 | 一比一模式 | 提炼模式 |
|------|-----------|---------|
| **Wiki 总行数** | ≤ 800 行 | > 800 行 |

**判断规则**：
- **行数优先**：以 Wiki 总行数为主要判断依据
- 若总行数 ≤ 800 行 → **一比一模式**（无论章节数、时间线条目数、表格数量如何）
- 若总行数 > 800 行 → **提炼模式**

**模式确认**：必须在日志中输出模式确认：
```
[MODE] 内容模式判断：Wiki {X} 行 → {一比一模式/提炼模式}
```

**3.1 一比一还原原则（一比一模式专用）**

Wiki → HTML 应保持内容的 1:1 对应关系。HTML 输出应完整反映 Wiki 源文件的内容结构，允许格式转换但不允许内容逻辑性缺失。
- **禁止精简**：不得因"条目过多"而删减时间线、列表、表格等内容
- **禁止省略**：不得因"内容重复"或"不重要"而省略任何章节
- **禁止优化**：不得因"提升可读性"而合并、重组或删减原文内容
- **唯一允许的操作**：格式转换（Markdown → HTML）、样式调整、标签前缀清洗

**3.2 提炼模式原则（提炼模式专用）**

当 Wiki 内容超过阈值时，允许采用提炼模式，但必须遵循以下规则：

**核心内容（禁止精简）**：
- **知识图谱**：所有 Mermaid 代码块必须完整渲染，不得遗漏
- **时间线数据**：所有时间线条目必须完整保留，不得删减
- **Sources 映射表**：必须完整保留，不得删减

**允许的提炼操作**：
- **表格格式调整**：保留所有表格的完整数据，但可以调整格式（如合并单元格、调整列宽）
- **列表格式调整**：保留所有列表项，但可以调整格式（如转换为表格、调整层级）
- **内容重组**：允许在不改变原意的前提下重组章节顺序，但必须保留所有章节
- **密度优化**：正文章节密度目标 60-80%（基于 Wiki 源文件的正文内容行数，排除参考资料、核心素材库、Sources 映射表等非正文内容）
- **合并策略**：允许将多个相关章节合并为一个（但必须保留所有章节标题）

**禁止的提炼操作**：
- **禁止删除章节**：所有 `##` 标题必须保留，不得删除任何章节
- **禁止删除核心内容**：不得删除知识图谱、时间线条目、Sources 映射表
- **禁止过度精简**：精简后正文内容不得少于原文正文内容的 60%
- **禁止过度填充**：禁止用填充散文、重复标题等方式来"填满"内容

**提炼记录**：必须在 HTML 末尾添加提炼记录区块，格式如下：
```html
<section class="distill-record">
  <h2>提炼记录</h2>
  <table>
    <thead><tr><th>类型</th><th>Wiki 原始数量</th><th>HTML 数量</th><th>说明</th></tr></thead>
    <tbody>
      <tr><td>章节</td><td>{X}</td><td>{Y}</td><td>{完整/精简说明}</td></tr>
      <tr><td>知识图谱</td><td>{X} 个</td><td>{Y} 个</td><td>完整渲染（禁止精简）</td></tr>
      <tr><td>时间线</td><td>{X} 条目</td><td>{Y} 条目</td><td>完整保留（禁止精简）</td></tr>
      <tr><td>表格</td><td>{X} 个</td><td>{Y} 个</td><td>{完整保留/格式调整}</td></tr>
      <tr><td>Sources映射表</td><td>1</td><td>1</td><td>完整保留（禁止精简）</td></tr>
    </tbody>
  </table>
</section>
```

**3.3 标题提取规则（两种模式通用）**
   - **从 Wiki 内容提取**：标题必须基于 Wiki 源文件的实际内容，而非文件名
   - **清洗标签前缀**：若 Wiki `#` 标题或文件名含标签前缀（如 `标签@科技史话_AI战争`），必须清洗，提取有意义部分（`AI战争`）
   - **有意义且符合内容**：标题应简洁明了地概括 Wiki 内容主题，便于用户理解
   - **标题格式**：可采用"分类：主题"结构，如"新消费观察：奢侈品市场深度分析"
   - **示例**：
     - 文件名：`标签@观察_新消费_奢侈品` → 标题："新消费观察：奢侈品市场深度分析"
     - 文件名：`标签@科技史话_AI战争` → 标题："科技史话：AI战争与人工智能发展"
     - 文件名：`标签@观察_商业历史_外卖大战` → 标题："商业历史：2025年中国外卖大战"
     - 无明确主题：从 Wiki 摘要或核心内容中提炼有意义标题

3. **内容提取与映射**：从 `wiki/{主题}.md` 中提取内容，映射到对应区块。**禁止使用与 Wiki 内容无关的通用兜底文案**（如"基于维基条目编译的长文档排版输出"）。允许在不改变原意的前提下重组或适配语气：
   - Wiki 标题（第一个 `#`） → 封面主标题（若含标签前缀则清洗，提取有意义部分）
   - Wiki 头部摘要 → 封面副标题 + Lead 段（可精简，但必须基于原文）
   - 文件名 `wiki/{主题}.md` → 封面定位信息（使用实际文件名，不得臆造）
   - **所有**知识图谱（可能有多个） → 对应的 Mermaid 代码块（或 `--svg` 时转为内联 SVG）
   - 人物/组织/时间线 → 正文章节（时间线数据可转换为 `tools/template/lite-diagrams.md` 中的 Timeline 图表作为可视化补充）
   - 核心数据/财务表 → 样式化表格（仅调整 HTML 格式）（若适合，可转换为 `tools/template/lite-diagrams.md` 中的 Donut/Bar/Waterfall 等图表作为可视化补充）
   - Sources（映射表）→ 底部来源区块（**完整保留**）

4. **Mermaid 处理**：
   - **完整渲染**：Wiki 中的**每一个** mermaid 代码块都必须渲染，不得遗漏
   - **渲染模式**：
     - `--lite` 模式默认：将 Mermaid 语法转为内联 SVG（零 JS 依赖，不加载 CDN）
     - `--lite --mermaid`：使用 Mermaid CDN 加载（加载 JS）
     - `--kami` 模式默认：使用 Mermaid CDN 加载
     - `--kami --svg`：将 Mermaid 转为手绘 SVG（不依赖 JS CDN）
   - **复杂度降级规则**（`--lite` 模式专用）：
     - 若 Mermaid 图谱过于复杂（如超过 20 个节点、多层次嵌套、复杂样式），AI 无法准确转为内联 SVG 时，**自动降级为 Mermaid CDN 方案**
     - 降级后必须在输出报告中告知用户：`⚠️ 已自动切换为 Mermaid CDN 模式（图谱过于复杂，无法准确转为内联 SVG）`
   - **自查清单**：生成 HTML 后，统计 Wiki 中的 mermaid 数量和 HTML 中的 mermaid 数量，确保一致
   - **尖括号转义规则**：Mermaid v11 会将节点标签中的 `<文本>` 误解为 HTML 标签并报 syntax error。生成 Mermaid 代码时，必须移除节点标签内的尖括号：
     - `["OpenAI<br/><估值8520亿$>"]` → `["OpenAI<br/>估值8520亿$"]`
     - `["马斯克<br/><xAI创始人>"]` → `["马斯克<br/>xAI创始人"]`
     - 规则：`<标签>` 中的 `<` 和 `>` 直接删除，保留内部文本

5. **SVG 图表自动选用（无条件生效）**：此规则不依赖 `--svg` 标志，在所有模式下均自动执行。当 Wiki 正文中的**表格、列表等结构化数据**（不含 Mermaid 代码块，Mermaid 由规则 4 处理）适合可视化时，主动从 kami `CHEATSHEET.md` 的 14 种内联 SVG 图表组件中选用最匹配的类型，嵌入为 `<figure>` 容器。选用依据：

   | 数据特征 | 推荐图表 | 典型场景 |
   |----------|----------|----------|
   | 含 OHLC/逐日价格 | Candlestick（K线） | 金价走势 |
   | 正负贡献求和 | Waterfall（瀑布图） | 收入桥、利润分解 |
   | 单系列占比 ≤6 项 | Donut（环形图） | 消费结构、需求分布 |
   | 多系列跨时间 | Line Chart（折线图） | 趋势对比 |
   | 分类对比 | Bar Chart（柱状图） | 品牌营收对比 |
   | 2×2 定位 | Quadrant（象限图） | 战略定位矩阵 |
   | 时间轴+里程碑 | Timeline（时间线） | 业绩时间线、历史节点 |
   | 层级关系 ≥2 层 | Tree（树状图） | 组织架构、品牌体系 |
   | 跨角色流程 | Swimlane（泳道图） | 业务流程 |
   | 流程决策分支 | Flowchart（流程图） | 决策逻辑 |
   | 分层架构 | Layer Stack（分层图） | 技术架构 |
   | 集合交集 2-3 组 | Venn（维恩图） | 用户重合度 |

   **优先级**：SVG 图表 > Mermaid > 纯表格。当数据同时适合表格和图表时，保留表格作为精确数据源，图表作为可视化补充。
   **密集数据处理**：当结构化数据条目过多（如 50+ 个时间线事件、20+ 个分类项），SVG 无法清晰展示时，应从全量数据中选取 8-10 个关键条目生成 SVG 图表作为全局概览，完整数据保留为 HTML 表格/列表作为精确记录。两者并存，互为补充。

   **图表选用规则**：
   - **Mermaid 图谱**：适用于 Wiki 中已有的 Mermaid 代码块（知识图谱、流程图等），必须完整渲染
   - **SVG 图表**（`tools/template/lite-diagrams.md`）：适用于 Wiki 中的结构化数据（表格、列表等），用于可视化补充
   - **两者并存**：当 Wiki 中既有 Mermaid 代码块，又有结构化数据时，两者同时存在，互为补充

   **Timeline 选择规则**：Wiki 中的时间线数据存在两种渲染方式，选择规则如下：
   - **SVG Timeline**（`tools/template/lite-diagrams.md`）：适用于**可视化概览**，选取关键里程碑（≤12 条目），作为全局视图
   - **HTML Timeline**（`tools/template/lite.html` 内置样式）：适用于**完整数据展示**，保留所有条目，作为详细记录
   - **必须同时保留**：SVG Timeline 只是可视化概览，不能替代完整数据。必须同时保留完整的 HTML 时间线表格，两者并存，互为补充

   **ASCII Art 转换规则**：Wiki 中以 ` ``` ` 或 ` ```text ` 包裹的 ASCII art 图表（如矩阵、关系图、树状结构），必须识别其结构并转换为 SVG 图表或语义化 HTML。不得直接将 ASCII art 原样输出为 `<pre>` 块。

   | ASCII art 结构特征 | 转换目标 | 典型场景 |
   |-------------------|----------|----------|
   | 矩阵/网格布局（行×列对齐） | HTML `<table>` 或 Quadrant SVG | 核心玩家矩阵、阵营对比 |
   | 框线+箭头关系图（┌──┐ + ──►） | Tree SVG 或 Mermaid graph | 关键人物关系图、权力结构 |
   | 缩进树状结构（├── └──） | Tree SVG 或嵌套 `<ul>` | 产业链机会、层级架构 |
   | 流程图（A → B → C） | Mermaid flowchart 或 Flowchart SVG | 决策流程、演进路径 |

   **转换原则**：
   - 先理解 ASCII art 传达的信息结构，再选择最合适的可视化方式
   - 转换后保留原图中的所有实体名称和关系标注
   - 若 ASCII art 过于复杂（如嵌套超过 3 层、实体超过 15 个），可降级为语义化 HTML 表格
   - 转换后的 SVG/HTML 必须在 780px 宽度内可读

6. **溯源保留**：Wiki 中的所有 `[本地快照](http://localhost:7026/reading/...)` 链接必须完整保留
7. **元数据填充**：自动填充 HTML `<head>` 中的 title、description、keywords

#### Step 4: 输出文件

1. **写入 HTML 文件**：根据用户在 Step 1 选择的输出方式，写入对应的文件路径

2. **写入后自查**：读取生成的 HTML 文件，执行以下自查流程：

   **自查步骤 1：内容模式验证**
   - 确认日志中输出了模式确认：`[MODE] 内容模式判断：...`
   - **验证**：必须有明确的模式声明

   **自查步骤 2：一比一还原验证（一比一模式专用）**
   - 统计 Wiki 源文件的章节数（`##` 标题数量）
   - 统计 HTML 中的章节数（`<section>` 或 `<h2>` 数量）
   - 统计 Wiki 时间线条目数（表格行数）
   - 统计 HTML 时间线条目数
   - **验证**：章节数必须 100% 一致，时间线条目数必须 100% 一致

   **自查步骤 3：提炼模式验证（提炼模式专用）**
   - 检查 HTML 末尾是否有提炼记录区块（`<section class="distill-record">`）
   - 验证提炼记录中的数量与实际 HTML 数量一致
   - 验证所有 `##` 标题都已保留（章节数必须 100% 一致）
   - **验证**：提炼记录存在、数量一致、章节完整

   **自查步骤 4：标题验证**
   - 检查 HTML `<h1>` 标题是否已清洗标签前缀
   - **验证**：标题不得包含 `标签@` 前缀，必须有意义且符合内容

   **自查步骤 5：技术规范验证**
   - **Mermaid**：若 Wiki 源文件含 Mermaid 代码块且使用了 CDN，`<head>` 中必须包含 `<script src="https://cdn.jsdelivr.net/npm/mermaid@11/dist/mermaid.min.js">` 和 `mermaid.initialize({startOnLoad:true,securityLevel:"strict"})`（标准 `<script>` 标签，非 ESM import）
   - **SVG 纯文本**：SVG `<text>` 元素中不得包含 markdown 语法（`**粗体**`、`[链接](url)`、`- 列表` 等），SVG 不渲染 markdown，会原样显示为文本
   - **列表格式**：Wiki 中的 `- 条目` 格式必须转换为 `<ul><li>` HTML 结构，不得将多个条目塞入单个 `<p>` 标签
   - **内容去重**：同一段摘要不应在封面、Lead、正文中重复出现超过 2 次
   - **章节结构**（`--kami`）：正文章节应使用 `<section class="chapter">` 包裹，确保打印分页
   - **暗色模式**（`--lite`）：toggle 按钮存在且可正常切换
   - **占位符**：不存在残留的 `{{...}}`

   **自查步骤 5a：SVG 图表选用验证**
   - 扫描 Wiki 源文件中的结构化数据（表格、列表），判断是否有适合可视化的数据未被转换为 SVG 图表
   - 参照 `tools/template/lite-diagrams.md` 的 8 种组件选用规则，检查是否遗漏
   - **验证**：适合可视化的数据已生成对应 `<figure>` 图表，纯表格数据和图表并存（图表为可视化补充，表格为精确数据源）

   **自查步骤 5b：ASCII Art 转换验证**
   - 检查 HTML 中是否存在残留的 ASCII art `<pre>` 块（含 `┌──┐`、`├──`、`──►`、矩阵对齐空格等特征）
   - **验证**：所有 ASCII art 代码块已转换为 SVG 图表或语义化 HTML，无 `<pre>` 原文残留

   **自查步骤 6：输出自查报告**
   - 必须输出自查报告，格式如下：

   **一比一模式自查报告**：
   ```
   📋 **自查报告：{主题}**（一比一模式）

   - 内容模式：一比一模式（Wiki ≤ 800 行，≤ 15 章节）

   ❌ **不通过项**（若有）：
   - {问题描述}

   🎯 **结论**：{通过 / 需要修正}
   ```

   **提炼模式自查报告**：
   ```
   📋 **自查报告：{主题}**（提炼模式）

   ✅ **通过项**：
   - 内容模式：提炼模式（Wiki {X} 行，{Y} 章节，{Z} 时间线条目）
   - 章节完整性：Wiki {X} 个章节，HTML {Y} 个章节（100%）
   - 提炼记录：{存在且数量一致 / 缺失}
   - 标题清洗：{已清洗或未清洗}
   - Mermaid 渲染：{数量} 个图表
   - SVG 图表：{数量} 个（适合可视化的数据已转换）
   ```

   **自查步骤 7：修正与重新写入**
   - 若自查报告中有任何"❌ 不通过项"，**立即修正**对应问题
   - 修正后重新写入 HTML 文件
   - **重新执行自查流程**，直到所有项都通过
   - **最多修正 3 次**：若 3 次修正后仍有不通过项，向用户报告问题并请求指导
6. 输出完成后，向用户报告：

### 5.1 `--lite` 工作流（所有入口共用）

当用户指定 `--lite` 时，按以下流程执行排版（内容已在 Step 1 中获取）：

#### Lite Step 1: 加载模板

1. 读取 `tools/template/lite.html` 作为排版骨架
2. 读取 `tools/template/lite-diagrams.md` 作为图表组件规范
3. 两者均位于项目根目录的 `tools/template/` 下

#### Lite Step 2: 内容编译

遵循 Step 3 的**一比一还原原则**和**标题提取规则**，排版目标为 `tools/template/lite.html` 的结构：

1. **Header**：
   - eyebrow：从 Wiki 文件夹名推导（`标签@观察_新消费_黄金` → "新消费观察 · 黄金专题"），无分类前缀则跳过
   - **标题**：使用 Wiki 源文件的 `#` 标题（**非文件名**）
   - 副标题/Lead：从 Wiki 摘要提取
   - meta：作者从当前项目顶层文件夹名称获取（如 `simpread-kb` → "SimpRead KB"），日期自动生成，描述固定"基于简悦 LLM Wiki Compiler 方案"
2. **Metric Cards**：从 Wiki 正文的"核心数据"表格提取 4 个关键指标，无数据则跳过
3. **TOC**：提取所有 h1 章节标题，生成 `<a href="#id">` 锚点链接
4. **知识图谱**：
   - 默认：将 Mermaid 语法转为内联 SVG
   - `--mermaid` 模式：用 `<pre class="mermaid">` + Mermaid CDN 加载
5. **正文**：Wiki 内容映射到 lite.html 的章节结构
6. **SVG 图表**：从 `tools/template/lite-diagrams.md` 的 8 种组件中选取，使用 lite.html 的 CSS token
7. **本地快照链接**：Wiki 中的 `[本地快照](http://localhost:7026/reading/4873)` 渲染为 `<a class="snapshot" href="..." target="_blank" rel="noopener">4873</a>`，链接文本仅保留快照 ID 数字，HTML 中直接写入 `target="_blank" rel="noopener"`
8. **Sources 映射表**：保留完整

#### Lite Step 3: 输出与自查

复用 Step 4 的输出与自查规则，但：
- 不检查 `<section class="chapter">`（lite 模式不使用此结构）
- 不检查 Mermaid CDN（除非使用 `--mermaid`）
- 额外检查：暗色模式 toggle 按钮是否存在

### 6. 排版规范摘要（`--kami` 模式专用）

以下规则继承自 kami，**仅在 `--kami` 模式下使用**。`--lite` 模式的排版规范见 `tools/template/README.md`。

- **设计语言**：暖色羊皮纸底（`#FDF8F0`）、墨蓝主色（`#1B365D`）、衬线字体层级
- **中文字体**：TsangerJinKai02（CDN 加载），fallback 为 Source Han Serif SC → Songti SC → Georgia
- **页面结构**：单栏、最大宽度 780px、A4 打印适配
- **表格**：深色表头、斑马纹行、hover 高亮
- **链接**：点线底部边框、hover 实线
- **Mermaid**：CDN 加载 mermaid v11，`securityLevel: "strict"`
